# 工程宪法 · Doc-First（文档先于代码）

> 本文件为项目**最高约束**，优先级高于 `spec/` 目录下所有契约文档。
> 当本文与任何 spec（含 `00-INDEX.md`）冲突时，以本文为准，并应立即修订冲突的 spec 以重新对齐。

## 第 1 条 核心法则（Doc-First）
**所有交互（任何对系统行为、数据模型、API、算法、UI 的改动或新增），必须先修改文档，再修改代码。**

- ❌ 禁止「先改代码跑通，再补文档」。
- ❌ 禁止把「代码已实现但文档尚未同步」作为交付状态。
- ✅ 唯一允许顺序：**文档变更（spec/harness）→ 评审/确认 → 代码变更 → 二者一致性校验**。

## 第 2 条 适用范围
- **后端**：实体/字段、录取与模拟算法、API 端点与契约、鉴权、错误码；
- **前端**：页面、角色视图、与 API 映射、交互行为；
- **数据**：种子数据、约束、派生值口径；
- **跨服务**：路由、数据隔离、一致性规则。
任何「实现偏离文档」的情形，均属本法调整范围。

## 第 3 条 标准工作流
1. **改文档**：在对应 spec 文档中描述目标行为/模型/接口，标注 `MUST/SHOULD/MAY`，并写入 `00-INDEX.md §0.5 修订记录`。
2. **评审**：文档变更需经确认（用户认可或自审留痕），先确定行为边界与验收标准，再动手写代码。
3. **改代码**：仅按已定文档实现，不得引入文档未描述的隐式行为。
4. **一致性校验**：改动后必须确认文档与代码一致（无残留旧表述、新概念已落地）。
5. **留痕**：代码改动对应文档修订记录行，明确「文档—代码一致性同步」。

## 第 4 条 与现有 Harness 的关系
- `00-INDEX.md §0.2` 的「偏离即修订」是本法在 spec 层的具体化；本法要求更前置——**不是「偏离再改」，而是「动手前先改」**。
- 任何 spec 修订，应首先反映本文的 Doc-First 精神。

## 第 5 条 例外与违规处理
- **紧急修复（hotfix）**：若生产缺陷必须先改代码止损，须在修复后 **24 小时内**补齐文档，并将该次「代码先行」记入 `00-INDEX.md §0.5 修订记录` 并标注 `EXCEPTION`。
- **纯重构/格式化**：不改变外部可观察行为（无 API/算法/模型变更）的代码整理，可仅更新必要的文档注释，不触发全流程，但仍不得使文档失实。
- **违规**：以「代码已改、文档未动」作为完成态交付的，视为违反宪法，应退回补齐文档。

## 第 6 条 验收门槛
- 代码改动若使文档失实，视为未完成（red）。
- 提测/交付前，`06-quality-harness.md` 的一致性检查必须包含「文档与代码一致」一项；本宪法是前置约束，不替代该检查。
